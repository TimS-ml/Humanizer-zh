# Subagent 原地编辑契约设计

## 背景

Humanizer-zh 当前主要面向单段文本和单份文档. 改写模式要求返回修订后的正文, 更改总结则是可选项. 这个约定不适合 subagent 原地编辑文件的场景.

一次批量 polish 中, 两个 subagent 返回了空结果, 也没有修改文件. 重派为更小的任务后才成功. 这类失败不一定来自 Skill 本身, 但现有约定留下了三个缺口:

- Caller 要求直接修改文件时, Skill 没有规定完成后必须返回什么.
- Skill 多处要求先询问作者. Subagent 通常无法直接向作者提问.
- Caller 提供的术语表, 保留清单和禁改范围与 Skill 默认规则冲突时, 优先级没有写明.

## 目标

这次只补齐行为契约. 不调整 24 种句子级模式, 不改技术写作深层检查, 也不引入 batch 文件数或行数限制.

## 设计

### Caller 约束优先

Caller 可以为当前任务指定术语表, 保留清单, 禁改范围和输出格式. 这些局部约束高于 Skill 默认行为. Skill 只处理 Caller 没有覆盖的部分.

直接修改文件的任务默认需要摘要. Caller 可以指定呈现形式. 只有 Caller 明确要求其他完成报告或要求静默时, 才替换或省略这些字段.

该规则不允许 Caller 绕过更高优先级的系统规则. 它只解决 Skill 内部默认值与当前任务要求之间的冲突.

### Subagent 不停下来提问

直接面向用户时, 遇到需要作者判断的缺口仍可提问. Subagent 或批量任务没有直接提问渠道时, 应保留原文并标记缺口. 可用标记包括 `[定义待补]`, `[机制待补]`, `[来源待补]` 和 `[待作者确认]`.

这一规则不授权 subagent 猜全称, 补来源或替作者选择立场.

### 区分两种改写交付

改写模式分为两种交付形式:

1. Return-text: Caller 要求返回修订正文. 输出完整修订文本. 更改总结仍可选.
2. In-place edit: Caller 要求直接修改文件. 默认在完成后返回摘要. 摘要至少包含文件名, 修改类别与数量, 行数变化, 以及未完成项或阻塞原因. Caller 明确要求其他完成报告或静默时, 按该要求执行.

若没有修改任何文件, 默认摘要必须明确写明原因. 未按 Caller 要求产生的空结果不算完成.

## 修改位置

功能实现只修改 `SKILL.md`. 本设计文档和对应实施计划会同步记录审阅后确定的契约边界.

- 在任务定义后增加 Caller 约束与 subagent 规则.
- 在处理流程中增加执行环境与交付形式判断.
- 将改写模式的输出拆成 return-text 与 in-place edit 两种契约.
- 在快速检查清单中增加 in-place edit 摘要检查.

`README.md` 暂不修改. 这些规则属于 agent 执行契约, 不改变用户侧的调用方式.

## 验证

- YAML frontmatter 保持合法, `name` 与 `allowed-tools` 不变.
- 现有审阅模式输出不变.
- Return-text 改写仍输出修订后的正文.
- In-place edit 默认要求非空摘要, 并保留 Caller 明确指定的例外.
- Subagent 无法提问时使用缺口标记, 不猜测内容.
- Caller 局部约束优先级写明, 且没有覆盖系统规则.
- 24 种句子级模式与三条现有硬约束保持不变.
- Git diff 只包含 `SKILL.md`, 本设计记录和对应实施计划.
